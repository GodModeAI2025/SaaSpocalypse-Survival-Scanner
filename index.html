Überblick

Was ist der SaaSpocalypse Survival Scanner?

Ein Claude Skill, der SaaS-Unternehmen auf sieben Dimensionen analysiert und einen gewichteten Disruption Score berechnet. Das Ergebnis: eine ehrliche, unterhaltsame Einschätzung, wie viel von einem Produkt heute schon ein KI-Prompt sein könnte — und was nicht.

🎯

Disruption Score (0–100)

Gewichteter Score über sieben Dimensionen — von Workflow-Komplexität bis Preismodell-Vulnerabilität. Kalibriert an echten Unternehmen wie Stripe, Salesforce oder Jasper AI.

📖

Keine trockene Analyse — eine charmant-satirische Geschichte über das Unternehmen. Faktisch fundiert, spezifisch auf das Produkt bezogen, und nie gemein.

Methodik

Wie wird der Score berechnet?

Sieben Dimensionen, unterschiedlich gewichtet. Jede Dimension bewertet einen Aspekt, der ein SaaS-Produkt vor KI-Disruption schützt — oder verwundbar macht.

Workflow-Komplexität

20%

Einfache Text-/Daten-Transformation? Hoch ersetzbar. Komplexe Multi-System-Orchestrierung mit Echtzeit-Abhängigkeiten? Schwer ersetzbar.

Daten-Moat

20%

Proprietäre Daten, die nicht replizierbar sind? Nutzerdaten, Netzwerkeffekte, historische Datasets — alles, was ein Skill nicht mitbringt.

Physische Infrastruktur

15%

Braucht das Produkt Hardware, Server, Echtzeit-Systeme? Eine .md-Datei kann kein DNS betreiben und keine Pakete ausliefern.

Regulatorischer Schutz

15%

Lizenzen, Zertifizierungen, Compliance-Anforderungen — regulatorische Burggraben, die kein Prompt überbrücken kann.

Integrations-Tiefe

15%

Wie tief ist das Produkt in Kunden-Workflows eingebettet? API-Ökosystem, Marketplace, Partner-Netzwerk — das schafft Wechselkosten.

UI/UX-Abhängigkeit

10%

Braucht das Produkt eine komplexe, interaktive Echtzeit-UI? Oder ist es im Kern ein Text-Workflow mit hübscher Oberfläche?

Preismodell-Vulnerabilität

5%

Ist das Pricing offensichtlich überhöht für das, was geliefert wird? Seat-based Pricing für Dinge, die ein LLM trivial kann, ist ein Warnsignal.

Score-Kategorien

Von Unantastbar bis War nie nötig

Der Disruption Score reicht von 0 (absolut unersetzbar) bis 100 (war eigentlich immer nur ein Prompt). Sieben Kategorien ordnen das Ergebnis ein.

0–10 🏛️ Unantastbar Physische Infrastruktur, Regulierung oder fundamentale Internet-Schicht. Kein .md-File kann das ersetzen.
11–25 🏰 Festung Starke Moats: Daten, Netzwerkeffekte, tiefe Integration. Sehr schwer zu ersetzen.
26–40 😰 Ins Schwitzen Verwundbar in Teilen. Kern-Moat existiert, aber AI knabbert an den Rändern.
41–60 🫠 Wackelkandidat Signifikante Teile des Produkts sind durch einen Skill abbildbar. Differenzierung schrumpft.
61–80 📉 Auf der Kippe Der Großteil des Produkts ist ein Workflow, den ein LLM direkt ausführen kann.
81–95 🫧 Fast weg Das Produkt ist im Wesentlichen eine UI um einen Prozess, den Claude besser kann.
96–100 🪄 War nie nötig Das Produkt hätte nie existieren sollen. Es ist buchstäblich ein Prompt.

· Die Story

Stellt euch vor: 2008, ein kleines Startup namens GitHub tritt an und sagt dem Entwickler-Universum: „Hey, wir machen Git hübsch." Was klingt wie eine Semesterarbeit, wurde zum digitalen Gravitationszentrum der gesamten Software-Industrie.

Heute leben auf GitHub über 100 Millionen Entwickler, 420 Millionen Repositories, und der Großteil aller Open-Source-Projekte, die das Internet am Laufen halten. Microsoft hat 2018 mit 7,5 Milliarden Dollar zugegriffen — und seitdem GitHub Actions, Copilot, Codespaces und ein komplettes DevOps-Ökosystem draufgepackt.

Der Witz an GitHub ist: Die eigentliche Technologie — Git — ist Open Source und kostenlos. Jeder kann einen Git-Server betreiben. GitLab, Bitbucket, Gitea — es gibt Alternativen. Aber niemand geht hin. Weil GitHub nicht Git ist. GitHub ist der Ort, wo 100 Millionen Entwickler bereits sind. Netzwerkeffekte sind ein Moat, den kein Skill der Welt graben kann.

Wo Claude allerdings an den Rändern knabbern kann: Issues schreiben, Pull Request Reviews, README-Generierung, Documentation, Wiki-Pflege, CI/CD-Konfiguration — das sind alles Text-Workflows, die ein LLM direkt ausführen kann. GitHub weiß das natürlich und hat Copilot als Antwort. Clever: Wenn KI dein Geschäft bedroht, werde selbst zur KI.

GitHub ist kein SaaS, das man durch einen Skill ersetzt. GitHub ist Infrastruktur. Wie Strom. Man kann das Licht besser nutzen — aber man ersetzt nicht das Kraftwerk.

· Claude sagt…

✅ Was ich ersetzen kann
README.md-Dateien schreiben, die tatsächlich jemand liest
Issue-Templates, Pull Request Descriptions und Changelogs generieren
GitHub Actions Workflows konfigurieren und debuggen
Code Reviews durchführen und Verbesserungen vorschlagen
Documentation und Wiki-Seiten erstellen und pflegen
CODEOWNERS, .gitignore und Repo-Konfiguration optimieren
❌ Was ich nicht ersetzen kann
Ein Git-Repository hosten — ich bin kein Server
100 Millionen Entwickler-Profile und deren Commit-History
GitHub Actions Runner — CI/CD braucht echte Compute-Infrastruktur
Pull Requests mit Branch Protection, Status Checks und Merge Queues
Das Social-Graph: Stars, Forks, Followers, Contributions-Heatmap
GitHub Packages, Container Registry, Dependabot-Ökosystem
„Ich bin ziemlich gut darin, die Texte rund um Code zu schreiben — READMEs, Issues, Docs, Reviews. Aber GitHub selbst? Das ist Infrastruktur mit 100 Millionen Nutzern. Ich kann die Deko machen, aber ich bin nicht das Gebäude."

· Analyse im Detail

7 Dimensionen · Gewichteter Durchschnitt = 18/100

Lesehinweis: Ein niedriger Score (z.B. 5) bedeutet, dass diese Dimension das Unternehmen schützt — schwer durch KI zu ersetzen. Ein hoher Score (z.B. 60) bedeutet hohe Verwundbarkeit.

· Replacement SKILL.md

--- name: github-dev-assistant description: >- Entwickler-Workflows rund um GitHub optimieren. README, Issues, PRs, Actions, Docs — alles was Text ist. Ersetzt NICHT GitHub selbst. --- # GitHub Dev Assistant Skill ## Purpose Automate text-heavy GitHub workflows: write READMEs, craft issues, review PRs, configure Actions, maintain documentation. Everything around the code — not the code hosting itself. ## Instructions ### What This Skill Does 1. README Generation — From repo structure to polished README.md with badges, TOC, and examples 2. Issue & PR Templates — Standardized, clear, actionable templates for any project 3. GitHub Actions — Write, debug, and optimize CI/CD workflows in YAML 4. Code Review — Analyze diffs, suggest improvements, explain changes 5. Documentation — Generate and maintain docs, wikis, changelogs, and CONTRIBUTING guides 6. Repo Hygiene — .gitignore, CODEOWNERS, branch protection recommendations ### What NOT To Do - Do NOT pretend you can host a Git repository - Do NOT claim you can replace GitHub's social graph (100M devs don't fit in a .md file) - Do NOT suggest alternatives to GitHub without acknowledging the network effect lock-in - Do NOT attempt to run CI/CD pipelines (you are text, not compute) ## Status: PARTIALLY POSSIBLE Claude handles the text layer beautifully — but GitHub is infrastructure, not a text workflow. The building stays; we just redecorate the lobby.

Dieser Skill-Entwurf zeigt, wie ein Claude Skill die Text-Workflows rund um GitHub optimieren könnte.
